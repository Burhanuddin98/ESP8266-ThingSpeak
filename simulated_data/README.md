# Simulated reading database — IoT MEMS wireless inclinometer

Synthetic sensor readings for the ESP8266 + MPU-6050 + ThingSpeak shoring-wall
inclinometer described in *"Monitoring of inclinations of shoring walls for
safety of substructure excavation work using MEMS based wireless
inclinometers."* Generated by `generate_dataset.py`.

The data reproduces what the **corrected firmware** (`Inclinometer/Inclinometer.ino`
in the ESP8266-ThingSpeak repo) publishes to ThingSpeak every 15 s, for the full
9-node / 65-day campaign, with statistics constrained to the paper's own numbers.

## What the setup produces

Each node is one ESP8266 NodeMCU + MPU-6050. The firmware fuses the
accelerometer tilt and integrated gyro rate through a complementary filter and
posts four ThingSpeak fields every 15 s:

| Field  | Meaning                                   | Unit |
|--------|-------------------------------------------|------|
| field1 | fused roll  = net inclination about X     | deg  |
| field2 | fused pitch = net inclination about Y     | deg  |
| field3 | accelerometer-only roll (unfiltered ref)  | deg  |
| field4 | die temperature                           | °C   |

`field3` is deliberately noisier and spikier than `field1` — it is the raw
accelerometer estimate before gyro fusion, included so the value of the
complementary filter is visible in the data.

## Deployment modelled (from the paper)

- **9 nodes.** Western wall (55 m × 10 m): W1–W5, ~7.5 m spacing. Northern wall
  (35 m × 10 m): N1–N4, ~7.0 m spacing.
- **65-day** monsoon campaign, **15 s** sampling → **5 760 readings/node/day**,
  **3 369 600 readings** total.
- **2° operational warning threshold.** Values are **net** inclination (the
  baseline no-load reading is already subtracted, as in the paper).
- Drilling-adjacent nodes (W3, W4, N3) respond most; tree-shielded nodes (W5, N4)
  respond least — matching the paper's discussion.

## Database schema (`inclinometer_sim.sqlite`)

```
sensors(sensor_id, name, wall, position_m, distance_to_drill_m,
        drill_coupling, retained_soil_m, tree_shielded, thingspeak_channel)

readings(id, sensor_id, created_at, entry_id,
         field1_roll_deg, field2_pitch_deg,
         field3_accel_roll_deg, field4_temp_c, threshold_exceeded)
        -- index on (sensor_id, created_at)

daily_summary(sensor_id, day, date, rainfall_mm, cum_rainfall_mm, temp_c,
              mean_incl_deg, max_incl_deg, min_incl_deg, pct_over_threshold)
```

## Query examples

```sql
-- All readings for node W3 on the first day
SELECT created_at, field1_roll_deg, field4_temp_c
FROM readings r JOIN sensors s ON s.sensor_id = r.sensor_id
WHERE s.name = 'W3' AND created_at LIKE '2023-06-01%';

-- Fraction of raw samples over the 2 deg threshold, per node
SELECT s.name, ROUND(100.0*AVG(threshold_exceeded),2) AS pct_over
FROM readings r JOIN sensors s ON s.sensor_id = r.sensor_id
GROUP BY r.sensor_id ORDER BY pct_over DESC;

-- Daily mean inclination time series for the western wall (for plotting / ML)
SELECT s.name, d.day, d.mean_incl_deg, d.rainfall_mm
FROM daily_summary d JOIN sensors s ON s.sensor_id = d.sensor_id
WHERE s.wall = 'Western' ORDER BY s.name, d.day;
```

The 15 s per-node series is directly usable as input for the LSTM/GRU forecasting
the paper lists as future work.

## How the data is constructed

For each node a **daily-mean net-inclination curve** is built from three physical
drivers, then anchored to the paper:

1. **Excavation/drilling response** — per-day intensity (ramp-up, weekday
   schedule with rest days, taper), scaled by a per-node `drill_coupling` that
   folds in distance-to-drilling and tree shielding.
2. **Rainfall response** — synthetic monsoon rainfall convolved with a ~3.5-day
   decaying kernel (pore-pressure build-up and dissipation).
3. **Structural creep** — slow cumulative wall movement as excavation deepens.

The curve is then offset so its **day 1–30 mean matches paper Table 1** and its
**day 31–65 mean matches paper Table 2**, with the transition ramped over ~3 days
(several nodes reverse sign between the two sets — W1: +1.31°→−1.31° — which a real
wall cannot do in one step). A narrow event nudges the campaign peak toward the
paper's reported value.

Within each day, the 15 s series adds a thermal diurnal tilt, complementary-filter
noise, and — during working hours — sparse drilling **impact transients** that
occasionally exceed the 2° threshold (as the paper reports).

## Honesty / caveats

- **Rainfall and temperature series are synthetic** (plausible Mumbai monsoon), not
  the paper's Table 3 values — transcribing 65×2 figures from the paper image was
  not worth the error risk. Temperature is held in the paper's 27–29 °C band.
- **Period means match Tables 1–2 to ≈0.01–0.08°.** The larger deviations
  (≈0.05–0.08°) are on the three sign-reversing nodes (W1, N1, N4), where ramping
  the transition necessarily borrows a little across the day-30 boundary. Exact
  matches are recoverable by hard-stepping the offset, at the cost of a physically
  impossible one-day jump. See `reconciliation.txt`.
- **Daily-mean peaks** track the paper's reported peaks within ~0.1–0.15°.
  **Instantaneous 15 s maxima** sit ~0.2–0.5° higher because they include drilling
  impact transients that the paper's daily/averaged plots smooth away.
- This is a **simulation for testing pipelines / dashboards / forecasting models**,
  not measured field data.

## Regenerate

```bash
python generate_dataset.py                 # default: 15 s, 65 days, seed 42
python generate_dataset.py --interval 60   # 1-min cadence -> ~4x smaller DB
python generate_dataset.py --days 30 --seed 7
```

Requires `numpy` and `matplotlib` (plot is best-effort). Runtime ≈20 s at 15 s
cadence on the reference machine; the SQLite file is ~345 MB.
